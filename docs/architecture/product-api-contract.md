# Product API Contract

## Purpose

This document defines the route-level API contract for the Product Dashboard, administration workflows, content review, recommendations, pricing, budgets, and audit access.

It does not define the OTLP ingestion contract. Codex telemetry ingestion is defined in [Codex Production Ingestion Contract](./codex-production-ingestion-contract.md).

## API Principles

- Product API routes are versioned under `/api/v1`.
- Every non-health request is scoped to a Customer Organization.
- Authentication proves the caller identity, but product authorization is resolved by Product Role Mapping.
- Azure Container Apps built-in authentication may be used for frontend authentication, but it does not replace product authorization.
- Session metadata access is separate from captured content access.
- The API must not return raw captured content unless Content Capture Policy, redaction status, caller role, and scope all allow it.
- Mutating routes that can be retried by clients require an `Idempotency-Key` header.
- Security, policy, content, credential, recommendation, pricing, budget, export, and deletion changes create Governance Audit Events.
- Error responses use a problem-details style envelope.

## Request Context

Every authenticated request resolves an Authorization Context:

- Customer Organization.
- Product user.
- External identity tenant.
- Product roles.
- Product scopes.
- Data residency region.
- Request correlation ID.

The Customer Organization is resolved from the authenticated user session, setup profile context, route scope, or explicit tenant selector. The API must reject ambiguous tenant context instead of choosing a default.

Required request headers:

| Header | Required | Notes |
| --- | --- | --- |
| `Authorization` | Yes, except health | Bearer token or authenticated app session context |
| `X-Correlation-Id` | Recommended | Generated by the API when absent |
| `Idempotency-Key` | Mutating retryable routes | Required for credential, content decision, recommendation regeneration, pricing, and budget mutations |

## Grafana-Originated Filter Links

Grafana-originated links are untrusted browser navigation inputs.

Allowed query parameters from Grafana:

- `from`
- `to`
- `environment`
- `region`
- `harness`
- `model`
- `modelProvider`
- `hotspotType`
- `cacheBustCategory`
- `findingState`
- `signalType`
- `result`
- `rejectionReason`

Rules:

- Unknown Grafana query parameters must be ignored or rejected with `validation_failed`.
- Grafana query parameters must never be interpreted as authorization grants.
- Grafana links must not contain `sessionId`, `productUserId`, `credentialId`, `traceId`, `spanId`, `contentReferenceId`, raw repository paths, file paths, Blob Storage URIs, or return URLs.
- Product API must re-check Customer Organization, Product Role Mapping, scope, and route action before returning any linked data.
- Product API must not redirect to caller-supplied absolute URLs.
- Grafana links can pre-filter `/overview` and `/sessions` only in the first release.

## Error Model

The API returns consistent problem-details responses.

```json
{
  "type": "https://docs.product.local/problems/authorization-denied",
  "title": "Authorization denied",
  "status": 403,
  "detail": "The caller is not allowed to review captured content for this session.",
  "correlationId": "01jz7h3f4f0g3d2x8w9v1n0m6q",
  "code": "authorization_denied"
}
```

Standard error codes:

- `authentication_required`
- `authorization_denied`
- `tenant_context_required`
- `tenant_context_ambiguous`
- `not_found`
- `validation_failed`
- `content_not_available`
- `redaction_not_confident`
- `policy_denied`
- `idempotency_key_required`
- `conflict`
- `rate_limited`

## System Routes

### `GET /api/v1/system/health`

Returns process liveness. It must not require tenant context.

### `GET /api/v1/system/readiness`

Returns dependency readiness for Product Metadata Store, telemetry backends, content store, and recommendation dependencies. It must be protected or restricted to operational callers.

## Current User Routes

### `GET /api/v1/me`

Returns:

- Product user identity.
- Customer Organizations visible to the caller.
- Product roles and scopes.
- Feature flags and policy summaries relevant to dashboard rendering.

This route is the dashboard bootstrap contract.

## Customer Organization Routes

### `GET /api/v1/customer-organization`

Returns the active Customer Organization profile, policy summary, data residency region, connected identity tenants, and enabled product capabilities.

Required action: `tenant.read`.

### `PATCH /api/v1/customer-organization`

Updates product-owned Customer Organization settings.

Required action: `tenant.update`.

Audit required: yes.

## Overview Routes

### `GET /api/v1/overview`

Returns aggregate overview summaries for visible tenant, team, repository, harness, model, token, cost, cache, hotspot, ingestion, and platform-health dimensions.

Supported filters:

- `from`
- `to`
- `environment`
- `region`
- `harness`
- `model`
- `modelProvider`
- `hotspotType`
- `cacheBustCategory`
- `findingState`
- `signalType`
- `result`
- `rejectionReason`

The route must preserve the non-punitive posture. It must not return individual developer rankings, raw session timelines, raw evidence packets, raw content, content review queues, or telemetry-store links.

Required action: `overview.read`.

## Identity Routes

### `GET /api/v1/identity/tenants`

Lists connected identity tenants for the active Customer Organization.

Required action: `identity.manage`.

### `POST /api/v1/identity/tenants`

Registers an identity tenant connection record.

Required action: `identity.manage`.

Audit required: yes.

### `GET /api/v1/identity/role-mappings`

Lists Product Role Mappings.

Required action: `identity.manage`.

### `POST /api/v1/identity/role-mappings`

Creates a Product Role Mapping from an external user or group claim to product roles and scopes.

Required action: `identity.manage`.

Audit required: yes.

### `PATCH /api/v1/identity/role-mappings/{roleMappingId}`

Updates a Product Role Mapping.

Required action: `identity.manage`.

Audit required: yes.

## Harness Setup Routes

### `GET /api/v1/harness-setup/profiles`

Lists manual setup profiles for supported Coding-Agent Harnesses.

Required action: `harness_profile.manage`.

### `POST /api/v1/harness-setup/profiles`

Creates a setup profile for a harness, team, repository scope, policy version, and data residency region.

Required action: `harness_profile.manage`.

Audit required: yes.

### `POST /api/v1/harness-setup/profiles/{profileId}/credentials`

Issues a Scoped Ingestion Credential for a developer and setup profile.

Required action: `ingestion_credential.manage`.

Audit required: yes.

Response must show the secret value only once. Subsequent reads expose metadata only.

### `POST /api/v1/harness-setup/credentials/{credentialId}/rotate`

Rotates a Scoped Ingestion Credential.

Required action: `ingestion_credential.manage`.

Audit required: yes.

### `POST /api/v1/harness-setup/credentials/{credentialId}/revoke`

Revokes a Scoped Ingestion Credential.

Required action: `ingestion_credential.manage`.

Audit required: yes.

## Session Routes

### `GET /api/v1/sessions`

Searches sessions.

Supported filters:

- `from`
- `to`
- `environment`
- `region`
- `harness`
- `model`
- `modelProvider`
- `teamId`
- `repositoryId`
- `developerScope`
- `metricStatus`
- `hotspotType`
- `cacheBustCategory`
- `findingState`
- `signalType`
- `result`
- `rejectionReason`
- `recommendationState`

The API must preserve the non-punitive posture by avoiding people-ranking sort modes. Person-scoped filters are allowed only for self-view or authorized investigation.

Required action: `session.read_own` or `session.read_scoped`.

### `GET /api/v1/sessions/{sessionId}`

Returns the Session Investigation View summary:

- Session metadata.
- Metric quality markers.
- Token and cost totals.
- Harness and model context.
- Repository context.
- Token Hotspots.
- Recommendation summary.
- Content evidence availability states.
- Audit context.

Required action: `session.read_own`, `session.read_scoped`, or `session.investigate`.

### `GET /api/v1/sessions/{sessionId}/timeline`

Returns ordered turns, invocations, tool activity summaries, token observations, cache evidence, and hotspot references.

Required action: `session.investigate`.

### `GET /api/v1/sessions/{sessionId}/content-references`

Returns content reference metadata and redaction state. It does not return captured content bodies.

Required action: `session.investigate`.

### `GET /api/v1/sessions/{sessionId}/recommendations`

Returns visible recommendations for a session.

Required action: `recommendation.read`.

## Content Review Routes

### `GET /api/v1/content-review/items`

Lists content items requiring privileged review.

Supported states:

- `review_required`
- `redaction_failed`
- `discarded`
- `approved_excerpt`

Required action: `content_review.read`.

### `GET /api/v1/content-review/items/{contentReferenceId}`

Returns content review metadata and any already-approved bounded excerpt. It must not expose raw failed content unless a future explicit break-glass workflow is defined.

Required action: `content_review.read`.

### `POST /api/v1/content-review/items/{contentReferenceId}/retry-redaction`

Retries redaction using the current approved redaction policy.

Required action: `content_review.decide`.

Audit required: yes.

### `POST /api/v1/content-review/items/{contentReferenceId}/discard`

Discards the content item and keeps metadata and audit state.

Required action: `content_review.decide`.

Audit required: yes.

### `POST /api/v1/content-review/items/{contentReferenceId}/approve-excerpt`

Stores a manually approved bounded excerpt when policy permits it.

Required action: `content_review.decide`.

Audit required: yes.

## Recommendation Routes

### `GET /api/v1/recommendations/{recommendationId}`

Returns recommendation detail, rationale, evidence links, confidence, authority state, model or rule version, and visible content excerpts if permitted.

Required action: `recommendation.read`.

### `POST /api/v1/recommendations/regeneration-requests`

Requests asynchronous recommendation regeneration for a session, hotspot, repository, or time range.

Required action: `recommendation.regenerate`.

Audit required: yes.

The API returns a request record, not a synchronous LLM answer.

Required action: `pricing.manage`.

Audit required: yes.

### `POST /api/v1/pricing/basis/{pricingBasisId}/reject`

Rejects a pending pricing basis candidate.

Required action: `pricing.manage`.

Audit required: yes.

### `POST /api/v1/pricing/basis`

Creates a Customer Organization pricing override.

Required action: `pricing.manage`.

Audit required: yes.

## Budget Routes

### `GET /api/v1/budgets/policies`

Lists Non-Punitive Budget Alert policies.

Required action: `budget.manage`.

### `POST /api/v1/budgets/policies`

Creates a budget alert policy scoped to team, repository, harness, model, workflow, or Customer Organization.

Required action: `budget.manage`.

Updates a budget alert policy.

## Audit Routes

### `GET /api/v1/audit-events`

Queries Governance Audit Events.

Supported filters:

- `from`
- `to`
- `actorUserId`
- `scopeType`
- `scopeId`
- `eventType`
- `correlationId`

Required action: `audit.read`.

Each returned item includes tenant, actor, action, target resource, decision, timestamp, correlation ID, and non-sensitive evidence metadata. The API must not return raw prompt text, code content, command output, tool results, secrets, tokens, or connection strings in audit evidence metadata.

### `GET /api/v1/ingestion-rejections`

Queries tenant-scoped ingestion rejection records.

Supported filters:

- `from`
- `to`
- `reasonCode`
- `signalType`
- `correlationId`

Required action: `audit.read`.

Each returned item includes tenant, harness setup profile, scoped credential identifier when derivable, declared harness, signal type, request route, reason code, HTTP status, timestamp, correlation ID, audit event ID when present, and non-sensitive evidence metadata. The API must not return raw prompt text, code content, command output, tool results, secrets, tokens, connection strings, or raw OTLP payload bodies.

## Pagination

Collection routes use cursor pagination:

- `limit`
- `cursor`

Responses include:

- `items`
- `nextCursor`
- `totalEstimate` when cheap to compute

## Idempotency

The API stores idempotency keys for mutating retryable operations with:

- Customer Organization.
- Caller user.
- Route.
- Request hash.
- Resulting operation ID.
- Expiry time.

If the same key is reused with a different request hash, the API returns `409 conflict`.

## MVP Acceptance Criteria

- Product Dashboard can bootstrap caller identity and visible scopes from `/api/v1/me`.
- PlatformAdmin can manage identity tenants, Product Role Mappings, harness setup profiles, and Scoped Ingestion Credential lifecycle.
- Session Investigation View can load summary, timeline, hotspots, recommendation summaries, and content reference states without querying telemetry stores directly from the browser.
- SecurityReviewer can process `review_required` and `redaction_failed` content states without raw failed content being stored as a Captured Content Blob.
- EngineeringLead can view scoped team or repository sessions without people-ranking sort modes.
- Developer can view their own sessions and recommendations.
- Pricing changes require explicit review before affecting Estimated Token Cost.
- State-changing governance operations create Governance Audit Events.
